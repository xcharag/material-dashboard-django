    15 · Patrones de Diseño¶

    Este documento explica cómo está organizado el código y qué patrones de diseño usamos de forma consciente en el proyecto. No es un catálogo: son las decisiones concretas que tomamos para que la aplicación de escritorio y la web sean fáciles de leer, de probar y de cambiar sin romper nada.

    La aplicación de escritorio está organizada en capas¶

    La app de escritorio (EEGDesktopApp) sigue una arquitectura en capas. Cada capa tiene una única responsabilidad y solo conoce a la capa que tiene debajo, nunca al revés. Eso es exactamente lo que se busca con este estilo: que un cambio en una capa (por ejemplo, cambiar la base de datos o el modelo de IA) no obligue a tocar las demás.

    El paquete backend/ está dividido físicamente en esas capas:

    backend/
├── presentation/     ← habla con la interfaz (el puente con el HTML/JS)
│   └── api.py            clase Api
├── application/      ← orquesta los casos de uso
│   └── session_manager.py
├── domain/           ← la lógica propia del problema (sin librerías externas de I/O)
│   ├── predictor.py      estrategias de clasificación de emociones
│   └── feature_extractor.py
├── infrastructure/   ← el mundo exterior: hardware y base de datos
│   ├── eeg_connector.py  puerto serie / protocolo Neurosky
│   └── db_client.py      PostgreSQL con psycopg2
└── common/           ← utilidades transversales
    └── observer.py

    La idea, capa por capa:

    • Presentación — La clase Api es lo único que ve el frontend. Traduce las llamadas de JavaScript en operaciones sobre las capas de abajo y devuelve diccionarios listos para pintar. No sabe SQL ni sabe cómo se decodifica la señal del dispositivo.
    • Aplicación — SessionManager coordina el caso de uso "grabar una sesión": empieza, va guardando lecturas y cierra. Decide cuándo pasan las cosas, pero delega los detalles técnicos.
    • Dominio — Aquí vive lo que hace único al proyecto: convertir la señal en características (feature_extractor) y clasificar la emoción (predictor). Es código Python "puro", sin puertos serie ni conexiones a base de datos, por lo que se puede probar de forma aislada.
    • Infraestructura — Todo lo que toca el exterior: leer el puerto Bluetooth y escribir en PostgreSQL. Si mañana cambiamos de diadema o de motor de base de datos, el impacto se queda en esta capa.

    El sentido de las dependencias siempre apunta hacia abajo. presentation conoce a application, domain e infrastructure; domain no conoce a nadie de más arriba. Además, SessionManager no importa directamente el cliente de base de datos: se lo pasamos por el constructor (SessionManager(get_settings, db_client, data_dir)), de modo que la capa de aplicación depende de un colaborador que le inyectan, no de un módulo concreto. Eso hace trivial sustituirlo por uno falso al probar.

    La aplicación web sigue el esquema en capas propio de Django (rutas → vistas → modelos/ORM → base de datos), que ya viene dado por el framework, así que aquí nos centramos sobre todo en la app de escritorio, donde la organización la definimos nosotros.

    Facade — la clase Api¶

    El frontend de la app de escritorio es HTML y JavaScript; toda la lógica está en Python. El puente entre ambos mundos es una sola clase, Api (backend/presentation/api.py), y esa clase es un Facade en el sentido más literal del término: una fachada que esconde varios subsistemas detrás de una interfaz simple.

    Cuando el JavaScript llama window.pywebview.api.get_realtime_data(), no tiene idea de que por debajo hay un hilo leyendo un puerto serie, un buffer de señal cruda, una red neuronal corriendo y una base de datos remota. Solo pide "los datos de ahora mismo" y recibe un diccionario con la forma de onda, las potencias de banda, la emoción y el estado de la sesión. Lo mismo con start_session() o connect_device(): nombres cortos que ocultan trabajo complejo.

    Nos gusta este patrón aquí porque mantiene el frontend tonto a propósito. El día que cambiemos cómo guardamos las lecturas o cómo cargamos el modelo, el JavaScript no se entera: la fachada sigue exponiendo los mismos métodos.

    Adapter — EEGConnector¶

    La diadema Neurosky no habla en diccionarios de Python. Habla el protocolo binario ThinkGear: bytes de sincronía 0xAA, longitudes de payload, checksums y códigos que indican si lo que viene es atención, meditación, señal cruda o potencias de banda. Es una interfaz incómoda de usar directamente.

    EEGConnector (backend/infrastructure/eeg_connector.py) es el Adapter que traduce ese flujo de bytes a algo cómodo. Por dentro decodifica el protocolo en un hilo en segundo plano; por fuera ofrece get_latest(), que devuelve un diccionario con attention, meditation y las bandas ya interpretadas. El resto del programa nunca se entera de que existe un puerto COM. Si algún día se cambia la diadema por otro dispositivo, solo hay que escribir otro adaptador que exponga la misma interfaz.

    Observer — el flujo de lecturas¶

    Este es el patrón que más cambió el diseño para bien. Antes, guardar las lecturas funcionaba "tirando" (pull): la interfaz preguntaba cada 200 ms y, cada vez que había datos y una sesión activa, se lanzaba un hilo nuevo para insertar esa lectura en la base de datos. Con una sesión de varios minutos eso significaba crear y destruir cientos de hilos efímeros, algo caro e innecesario.

    Ahora usamos Observer (publicación–suscripción). EEGConnector es el sujeto observable: cada vez que termina de decodificar un cuadro completo de datos (atención, meditación y bandas), notifica a quien esté suscrito en lugar de esperar a que le pregunten. La utilidad Observable (backend/common/observer.py) guarda la lista de suscriptores y les entrega el dato; no sabe quiénes son, así que el productor y los consumidores quedan totalmente desacoplados.

    # El conector solo anuncia; no sabe quién escucha.
self._connector.subscribe(self._on_frame)

def _on_frame(self, packet):
    if not self._session.is_active:
        return
    reading = { ... }               # arma la lectura con la emoción actual
    self._session.add_reading(reading)   # la encola; un único hilo la persiste

    Dos ventajas concretas:

    1. El dato llega en cuanto existe, no en el siguiente tic del polling.
    2. Desaparecen los hilos por lectura. En vez de un hilo por inserción, SessionManager tiene un solo hilo trabajador permanente que consume una cola y escribe en la base de datos. Una ráfaga de lecturas ya no provoca una ráfaga de hilos.

    El conector no conoce a SessionManager, y SessionManager no conoce al conector: los une la suscripción. Añadir mañana otro consumidor (por ejemplo, un registro de auditoría) es cuestión de un subscribe() más, sin tocar nada de lo existente.

    Strategy (+ Factory Method) — la clasificación de emociones¶

    Para decidir la emoción tenemos dos algoritmos que pueden intercambiarse:

    • CNNPredictor — la red neuronal entrenada (TensorFlow).
    • HeuristicPredictor — una regla sencilla basada en potencias de banda y en los valores de atención/meditación del propio dispositivo.

    Antes, la elección entre uno y otro vivía como un if disperso en el código ("si el modelo ya cargó, usa la CNN; si no, usa la heurística"). Ahora los dos implementan la misma interfaz, Predictor (backend/domain/predictor.py), con el mismo método predict(raw_samples, packet) y el mismo diccionario de salida. Eso es Strategy: el que llama pide una predicción y le da igual qué algoritmo hay detrás.

    class Predictor(ABC):
    @abstractmethod
    def predict(self, raw_samples, packet) -> dict: ...

class CNNPredictor(Predictor): ...
class HeuristicPredictor(Predictor): ...

    Gracias a eso, el bucle en tiempo real se quedó en una sola línea, sin condicionales:

    self._last_emotion = self._predictor.predict(raw_buf, pkt)

    La elección de la estrategia la resuelve una pequeña fábrica, create_predictor() (un Factory Method): intenta construir la CNN y, si sus archivos no cargan, devuelve la heurística. Quien la usa recibe siempre un Predictor listo para funcionar sin saber cuál le tocó.

    def create_predictor(models_dir) -> Predictor:
    try:
        return CNNPredictor(models_dir)     # el modelo entrenado, si carga
    except Exception:
        return HeuristicPredictor()         # si no, la regla de respaldo

    Como TensorFlow tarda unos 30 segundos en importarse, la app arranca usando la heurística (para que la ventana abra al instante) y, cuando la fábrica termina de cargar la CNN en segundo plano, simplemente cambia la estrategia activa. Añadir un tercer modelo en el futuro sería crear otra clase Predictor y enseñarle a la fábrica a elegirla, sin tocar el resto del programa.

    Otros patrones presentes¶

    Además de los anteriores, que son los que aplicamos con más intención, en el código aparecen otros de forma natural: la herencia de plantillas de Django y las vistas basadas en clases son Template Method (el esqueleto lo pone el framework, nosotros redefinimos los pasos concretos); el decorador @login_required que envuelve las vistas sensibles es un Decorator de manual; y al recorrer los QuerySets del ORM estamos usando el patrón Iterator sin escribir una línea para ello. No los explicamos en detalle porque vienen dados por el framework, pero es útil reconocerlos.
