    15 · Patrones de Diseño¶

    Análisis del proyecto según el catálogo de patrones de diseño de Refactoring.Guru. Primero se identifican los patrones ya presentes en el código (propios o heredados del framework) y después se evalúa qué otros patrones valdría la pena aplicar, con una puntuación que considera la escala real del sistema: máximo ~10 usuarios simultáneos.

    Patrones ya presentes¶

    Patrón Tipo Dónde
    Facade Estructural Api (app de escritorio)
    Strategy (informal) Comportamiento EmotionPredictor CNN vs. heurística
    State (informal) Comportamiento SessionManager activa/inactiva
    Adapter Estructural EEGConnector (protocolo ThinkGear → dict)
    Template Method Comportamiento CustomLoginView, plantillas Django
    Decorator Estructural @login_required sobre las vistas
    Iterator Comportamiento QuerySets del ORM de Django
    Singleton (informal) Creacional Instancia única de Api, logger del conector

    Facade — backend/api.py¶

    La clase Api es un Facade de libro: el frontend JavaScript (window.pywebview.api.*) habla con una sola interfaz simplificada que oculta cuatro subsistemas — EEGConnector (hardware), EmotionPredictor (inferencia), SessionManager (grabación) y db_client (persistencia). El frontend nunca conoce hilos, buffers ni SQL: llama get_realtime_data() y recibe un dict listo para pintar.

    Strategy — backend/emotion_predictor.py¶

    Existen dos algoritmos intercambiables para clasificar la emoción:

    • EmotionPredictor.predict() — la CNN entrenada (TensorFlow).
    • EmotionPredictor.heuristic() — regla basada en bandas de potencia y eSense.

    api.py elige en tiempo de ejecución: mientras TensorFlow carga (~30 s) usa la heurística y luego conmuta a la CNN. Ambas devuelven la misma estructura de salida ({label, confidence, probs, color, estimated}), que es la esencia del patrón. Es una implementación informal (un if en lugar de una interfaz común), ver recomendaciones abajo.

    State — backend/session_manager.py¶

    SessionManager cambia su comportamiento según su estado interno (_active): start() rechaza iniciar si ya hay sesión, add_reading() ignora lecturas sin sesión activa y stop() falla si no hay nada que cerrar. Es el patrón State en su forma mínima (bandera + guardas en lugar de clases-estado), suficiente para dos estados.

    Adapter — backend/eeg_connector.py¶

    EEGConnector adapta una interfaz incompatible: el protocolo binario ThinkGear (bytes de sincronía 0xAA, checksums, códigos de payload) se convierte en una interfaz Python amigable — get_latest() devuelve un dict con attention, meditation y bandas. El resto de la aplicación no sabe que existe un puerto serie.

    Template Method — vistas y plantillas de Django¶

    • CustomLoginView(auth_views.LoginView) redefine pasos concretos de un algoritmo cuyo esqueleto (validar formulario → autenticar → redirigir) vive en la clase base de Django.
    • La herencia de plantillas ({% block extrastyle %}, {% block extra_js %}) es el mismo patrón aplicado al renderizado: el layout base define el esqueleto y cada página redefine solo sus bloques.

    Decorator — @login_required¶

    Cada vista sensible se envuelve con @login_required (y las de IA con la verificación de rol), añadiendo el control de acceso sin modificar la función original. Es la variante funcional de Python del patrón Decorator.

    Iterator y Singleton (aportados por el entorno)¶

    • Los QuerySets del ORM se recorren sin exponer su estructura interna (Iterator), y además son lazy (rasgo de Proxy virtual).
    • La app de escritorio crea una única instancia de Api para toda la ventana, y el logger del conector se configura una sola vez (if not logger.handlers): Singletons informales a nivel de módulo.

    Patrones que valdría la pena aplicar (ranking)¶

    Puntuación sobre 10 = beneficio real ÷ esfuerzo, penalizando la sobre-ingeniería: con ≤10 usuarios simultáneos, un patrón solo vale si mejora la mantenibilidad o corrige un problema concreto, no por escalabilidad.

    # Patrón Puntuación Dónde aplicarlo
    1 Observer 8/10 Flujo de lecturas EEG en la app de escritorio
    2 Strategy (formalizado) 7/10 Interfaz común Predictor para CNN y heurística
    3 Singleton / caché de settings 6/10 SessionManager relee settings.json en cada lectura
    4 Factory Method 6/10 Creación del predictor según disponibilidad de TensorFlow
    5 Builder 5/10 Construcción del contexto clínico para el RAG
    6 Command 4/10 Operaciones del calendario (editar/cancelar/reprogramar)
    7 Proxy (caché) 3/10 Lista de pacientes en la app de escritorio
    8 Memento 3/10 Deshacer cambios en el calendario

    1 · Observer — 8/10¶

    Hoy el flujo de datos es pull: el frontend hace polling cada 200 ms y api.py lanza un hilo nuevo por cada lectura para insertarla en la base de datos (threading.Thread(target=self._session.add_reading, ...)). Con Observer, EEGConnector notificaría a suscriptores (SessionManager, predictor, UI) cuando llega un paquete: se elimina la creación de ~5 hilos por segundo, se desacopla api.py de los detalles de cada consumidor y el dato llega en cuanto existe, no en el siguiente tick del polling. Es el patrón con mejor relación beneficio/esfuerzo porque corrige un costo real (hilos efímeros) además de mejorar el diseño.

    2 · Strategy formalizado — 7/10¶

    La selección CNN/heurística ya existe pero vive como if self._predictor: en api.py. Formalizarla con una interfaz común (CNNPredictor y HeuristicPredictor con el mismo método predict()) elimina el condicional, permite probar cada estrategia por separado y facilita añadir un tercer modelo en el futuro sin tocar api.py. Esfuerzo bajo (las dos implementaciones ya devuelven la misma estructura).

    3 · Singleton / caché de configuración — 6/10¶

    SessionManager.add_reading() llama a get_settings() — que lee y parsea settings.json del disco — con cada lectura (~5 veces por segundo durante toda la sesión). Un objeto de configuración único, cargado una vez e invalidado solo cuando el usuario guarda cambios, elimina cientos de lecturas de disco por sesión con un cambio mínimo.

    4 · Factory Method — 6/10¶

    Complemento natural del punto 2: una fábrica create_predictor(models_dir) decide qué estrategia instanciar (CNN si el modelo carga, heurística si no) y api.py deja de conocer los detalles de construcción de cada una. Juntos, 2 y 4 convierten el fallback actual en un diseño explícito y testeable.

    5 · Builder — 5/10¶

    _build_patient_context() en las vistas de IA arma un texto largo paso a paso (datos del paciente + consultas + notas + adjuntos). Un Builder (ClinicalContextBuilder.add_consultations().add_notes().build()) haría cada sección testeable y reutilizable (el mismo builder serviría para el PDF). Beneficio de legibilidad, no de rendimiento.

    6 · Command — 4/10¶

    Las operaciones del calendario (reprogramar por arrastre, cancelar, editar) podrían encapsularse como comandos con undo(), habilitando "deshacer" en la UI. Mejora de experiencia de usuario real, pero el esfuerzo es considerable y con pocos usuarios el riesgo de errores concurrentes es bajo.

    7–8 · Proxy y Memento — 3/10¶

    • Un Proxy con caché para get_patients() evitaría una consulta remota por apertura del selector; con ~10 usuarios y listas pequeñas, el ahorro es marginal.
    • Memento para restaurar estados del calendario es elegante pero costoso; Command (punto 6) cubriría el mismo caso de uso más barato.

    Patrones que NO conviene aplicar a esta escala¶

    Para un sistema de ≤10 usuarios simultáneos, estos patrones añadirían complejidad sin retorno:

    • Abstract Factory, Bridge, Visitor — no hay familias de productos ni jerarquías dobles que lo justifiquen.
    • Flyweight — optimiza memoria con miles de objetos repetidos; aquí no existe ese volumen.
    • Chain of Responsibility — el middleware de Django ya cumple ese rol en la web; duplicarlo en el código propio sería redundante.
    • Mediator — la clase Api ya centraliza la coordinación de la app de escritorio; formalizarlo solo añadiría una capa.

    La regla general aplicada en este ranking: un patrón se gana su lugar cuando elimina un problema observable en el código (hilos por lectura, lecturas de disco repetidas, condicionales de estrategia), no por anticipar una escala que el sistema no tendrá.
